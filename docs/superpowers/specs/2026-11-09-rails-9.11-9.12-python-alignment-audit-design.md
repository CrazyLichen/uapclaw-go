# 9.11-9.12 Rails Python 对齐审核修复设计

> 审核 ProgressiveToolRail、TaskCompletionRail、TaskPlanningRail、AgentModeRail
> 及关联 DeepAgent/CreateSubagent/factory 与 Python 实现的差异修复

## 审核结论

审核发现 20 项差异，确认修复 12 项，标记待回填 2 项，不改 6 项。

## 修复项清单

### 🔴 高严重度

#### Fix-1: CreateSubagent 删除多余工厂分支

**问题：** Go 多了 `plan_agent`/`verification_agent`/`explore_agent` 三个 stub 分支，
Python 中这些子 Agent 的 `factory_name` 为空走 default → `create_deep_agent`。

**修复：** 删除 3 个 stub case，让它们走 default 通用工厂。

**文件：** `harness/deep_agent.go` CreateSubagent 方法

---

#### Fix-2: AgentModeRail buildEnterPlanModeStatus/buildPlanFileInfo 三状态分支 + 提示词复刻

**问题：**
1. Go 用 `planExists`（文件是否存在于磁盘）判断，Python 用 `planFilePath 是否非空`（enter_plan_mode 是否已调用）判断
2. 提示词文本与 Python 完全不同（6+ 处），包括缺少工具名引用、中英混杂、紧迫语气缺失

**修复：**
- 改为三状态分支：
  - 状态①：`planFilePath == ""` → 未调用 enter_plan_mode
  - 状态②：`planFilePath != ""` && `!planExists` → 已调用但文件未创建
  - 状态③：`planFilePath != ""` && `planExists` → 已调用且文件已存在
- `buildEnterPlanModeStatus`：用"路径是否非空"判断是否已调用，提示词一比一复刻 Python
- `buildPlanFileInfo`：提示词一比一复刻 Python（含 `edit_file`/`write_file` 工具名引用）

**Python 提示词参考（_build_enter_plan_mode_status）：**
- 有路径(CN): `"enter_plan_mode 已调用完成。请继续工作流。"`
- 有路径(EN): `"enter_plan_mode has been called. Proceed with the workflow."`
- 无路径(CN): `"你尚未调用 enter_plan_mode。请立即调用它作为你的第一个操作。"`
- 无路径(EN): `"You have NOT called enter_plan_mode yet. Call it NOW as your first action."`

**Python 提示词参考（_build_plan_file_info）：**
- 无路径(CN): `"尚无 plan 文件。请先调用 enter_plan_mode 创建。"`
- 无路径(EN): `"No plan file yet. Call enter_plan_mode first to create one."`
- 路径+存在(CN): `"计划文件已存在于 {path}。你可以使用 edit_file 工具读取并增量编辑它。"`
- 路径+存在(EN): `"A plan file already exists at {path}. You can read it and make incremental edits using the edit_file tool."`
- 路径+不存在(CN): `"计划文件尚不存在。你应该使用 write_file 工具在 {path} 创建计划。"`
- 路径+不存在(EN): `"No plan file exists yet. You should create your plan at {path} using the write_file tool."`

**文件：** `harness/rails/agent_mode.go` buildEnterPlanModeStatus / buildPlanFileInfo

---

#### Fix-3: _is_task_tool_registered 对齐 Python 查询 AbilityManager

**问题：** Go 只查 `ownsTaskTool` flag，无法检测其他 Rail 注册的 task_tool。

**修复：** 改为查询 `agent.AbilityManager().Get("task_tool")` 或遍历已注册工具检查是否存在同名 task_tool。

**文件：** `harness/rails/agent_mode.go` isTaskToolRegistered

---

### 🟡 中严重度

#### Fix-4: TaskPlanningRail BeforeModelCall 早退逻辑对齐

**问题：** Python 在 `system_prompt_builder is None` 时整体 return，Go 仅跳过 prompt injection 仍执行 model switching。

**修复：** 在 BeforeModelCall 开头获取 builder，若 nil 则直接 return nil。

**文件：** `harness/rails/task_planning.go` BeforeModelCall

---

#### Fix-5: ProgressiveToolRail initVisibleTools 有序初始化

**问题：** Go 遍历 map 无序，Python 按插入序 `[*always_visible, *default_visible]`。

**修复：** initVisibleTools 中先收集 alwaysVisible 再收集 defaultVisible 到 `[]string`，然后 dedup，保证 always 在 default 前面。

**文件：** `harness/rails/progressive.go` initVisibleTools

---

#### Fix-6: ProgressiveToolRail search_tools trace 事件对齐 Python

**问题：** Go 用 `{"event_type": "tool_search", "count": N, "agent_id": "..."}`,
Python 用 `{"action": "search_tools", "detail_level": N, "match_count": N}`。

**修复：** 改为 Python 结构：`{"action": "search_tools", "query": ..., "limit": ..., "detail_level": ..., "match_count": ...}`。
去掉 `agent_id`，加上 `detail_level` 和 `match_count`。

**文件：** `harness/rails/progressive.go` searchTools 方法中 appendTrace 调用

---

#### Fix-7: AgentModeRail BeforeModelCall 操作顺序对齐

**问题：** 非 plan 模式下 Python 顺序 remove_section → sync_task_tool → filter_hidden，
Go 顺序 remove_section → filter_hidden → sync_task_tool。

**修复：** 调整为先 syncTaskToolForModelToolInputs 再 filterHiddenTools。

**文件：** `harness/rails/agent_mode.go` BeforeModelCall

---

### 🟢 低严重度

#### Fix-8: 中文引号统一为 ASCII 直引号

**问题：** ProgressiveToolRail 中文提示词用 Unicode 弯引号 `\u201c\u201d`，
Python 用 ASCII 直引号 `"`。

**修复：** 2 处改为 `"`：
1. `progressiveToolNavigationHeaderCN` 中 `"工具地图"` 和 `"全部可立即调用的工具清单"`
2. `progressiveToolRulesBodyCN` 规则 8 中 `"下一步我将……"`

**文件：** `harness/prompts/sections/progressive_tool_rail.go`

---

#### Fix-9: isPlanFile 用 filepath.EvalSymlinks 解析 symlink

**问题：** Python 用 `Path.resolve()` 解析 symlink，Go 用 `filepath.Abs()` 不解析。

**修复：** 改为 `filepath.EvalSymlinks(filepath.Abs(p))` 组合。

**文件：** `harness/rails/agent_mode.go` isPlanFile

---

#### Fix-10: TaskPlanningRail Init 工具去重

**问题：** Python 检查 AbilityManager 是否已有 todo 工具再注册，Go 每次都新建。

**修复：** Init 中遍历 `am.List()` 检查是否已有 `todo_create`/`todo_list`/`todo_get`/`todo_modify`，
已存在则跳过创建。

**文件：** `harness/rails/task_planning.go` Init

---

#### Fix-11: TaskPlanningRail resource_mgr 集成

**问题：** Init/Uninit 缺少 resource_mgr 注册/注销，Python 调用 `Runner.resource_mgr.add_tool()` / `remove_tool()`。

**修复：**
- Init：每个 todo 工具注册时额外调 `resourceMgr.AddTool(t)`
- Uninit：每个 todo 工具注销时额外调 `resourceMgr.RemoveTool(t.Card().ID)`

**文件：** `harness/rails/task_planning.go` Init / Uninit

---

#### Fix-12: DeepAgentInterface 加 Card() + buildAvailableAgents 对齐

**问题：**
1. `DeepAgentInterface` 缺少 `Card()` 方法，导致通过 `ReactAgent().Card()` 间接获取
2. `buildAvailableAgents` 非 SubAgentConfig 分支只调 `SpecName()`，不尝试读 Card

**修复：**
1. 在 `DeepAgentInterface` 接口声明 `Card() *agentschema.AgentCard`
2. 替换项目中所有 `ReactAgent().Card()` 为直接 `.Card()`（2 处）
3. `buildAvailableAgents` 非 SubAgentConfig 分支：类型断言 `BaseAgent`，调 `Card()` 获取 name/description，
   失败 fallback 到 `SpecName()` + "DeepAgent instance"，对齐 Python `getattr(spec, "card", None)` 逻辑

**文件：**
- `harness/interfaces/deep_agent.go` — 加 Card() 声明
- `harness/tools/agent_mode/task_tool_factory.go` — 替换 ReactAgent().Card()
- `harness/tools/subagent/task_tool.go` — 替换 ReactAgent().Card()
- `harness/rails/agent_mode.go` — buildAvailableAgents 修复
