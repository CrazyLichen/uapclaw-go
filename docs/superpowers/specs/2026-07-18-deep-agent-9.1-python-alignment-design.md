# DeepAgent 9.1 Python 对齐修复设计

## 概述

当前 Go DeepAgent 实现与 Python 参考实现存在多处逻辑不一致，涵盖 follow-up drain、退出检查、回调错误处理、流式取消保护等关键路径。本设计逐项对齐 Python 行为，确保 Go DeepAgent 的功能正确性。

## 修复项清单

### 1. runTaskLoop follow-up drain 完整逻辑（P0）

**问题**：`runTaskLoop` 中 `_ = ctrl.DrainFollowUp()` 丢弃返回值，follow-up 功能完全失效。

**Python 参考**（`deep_agent.py` L2026-2040）：
```python
new_follow_ups = controller.drain_follow_up()
_state = self.load_state(session)
if new_follow_ups:
    _state.pending_follow_ups.extend(new_follow_ups)
is_follow_up = bool(_state.pending_follow_ups)
if _state.pending_follow_ups:
    current_query = _state.pending_follow_ups.pop(0)
    self.save_state(session, _state)
```

**修复**：
1. `newFollowUps := ctrl.DrainFollowUp()` — 捕获返回值
2. `st := d.LoadState(sess)` → `st.PendingFollowUps = append(st.PendingFollowUps, newFollowUps...)` — 合并
3. `isFollowUp := len(st.PendingFollowUps) > 0` — 判断
4. 若 `isFollowUp`，`currentQuery = st.PendingFollowUps[0]`，`st.PendingFollowUps = st.PendingFollowUps[1:]` — 弹出
5. `d.saveState(sess, st)` — 保存
6. `SubmitRound` 传 `isFollowUp` 而非硬编码 `false`

**影响文件**：`deep_agent.go` L1740-1749

### 2. runTaskLoop 退出检查（P0）

**Python 参考**（`deep_agent.py` L2079-2086）：

```python
if (controller.has_follow_up() or _state.pending_follow_ups):
```

```go
if ctrl.HasFollowUp() || len(st.PendingFollowUps) > 0 {

```

**影响文件**：`deep_agent.go` L1793-1802

### 4. FireLifecycle AFTER 回调错误处理（P1）

**问题**：`_ = c.Fire(after)` 静默丢弃 AFTER 回调错误，与 Python 不一致。

**Python 参考**（`base.py` L377-416 `lifecycle()` 方法）：
- 有原始异常时：log AFTER 错误，返回原始异常
- 无原始异常时：返回 AFTER 错误

**修复**：对齐 `RailExecutor.fireAfter()` 的逻辑：
```go
afterErr := c.Fire(after)
if afterErr != nil {
    if origErr != nil {
        logger.Error(logComponent).Str("event", string(after)).Err(afterErr).
            Msg("after 回调出错，掩盖原始异常")
        return origErr
    }
    return afterErr
}
```

**影响文件**：`rail/context.go` FireLifecycle 方法

### 5. hotReconfigure sync_builder_to_active_rails（P1 — 暂不修复）

**问题**：Python 在 `_hot_reconfigure` 末尾调用 `_sync_builder_to_active_rails()`，Go 未实现。

```go
// 注：Python 在 hotReconfigure 后调用 _sync_builder_to_active_rails() 将新 builder
// 同步到已注册 rail。Go 的策略不同：agent 通过 SystemPromptBuilder() 接口返回最新
// builder，rail 调用接口获取，无需显式同步。若后续 rail 体系需要持有 builder 引用，
// 再补充 syncBuilderToActiveRails 逻辑。
```

**影响文件**：`deep_agent.go` hotReconfigure 方法

### 9. runTaskLoopStream context 取消处理（P2）

**问题**：`streamCancel` 从未赋值，`cancelStreamProcess()` 无效；context 取消时后台 goroutine 可能长期存活。

**Python 参考**（`deep_agent.py` L2196-2212）：
```python
task = asyncio.create_task(_stream_process())
self._stream_process_task = task
try:
    async for chunk in session.stream_iterator():
        yield chunk
    await task
except asyncio.CancelledError:
    await self._cancel_stream_process_task()
    raise
finally:
    if self._stream_process_task is task:
        self._stream_process_task = None
```

**修复**：
1. `streamCtx, streamCancel := context.WithCancel(ctx)` — 派生子 context
2. `d.streamMu.Lock(); d.streamCancel = streamCancel; d.streamMu.Unlock()` — 存储 cancel
3. 后台 goroutine 使用 `streamCtx` 而非 `ctx`
4. 前台 goroutine 退出时调用 `d.cancelStreamProcess()` 取消后台
5. `defer` 中清理 `d.streamCancel = nil`

**影响文件**：`deep_agent.go` runTaskLoopStream 方法

### 14. HandleInput 中 run_context 类型断言错误（P1）

**问题**：`SubmitRound` 将 `*RunContext` struct 存入 metadata，但 `HandleInput` 用 `v.(string)` 断言，永远失败。导致 executor 收不到 `run_context`，心跳场景下 ReActAgent 无法判断 lightweight context 模式。

**传递流程**：
```
SubmitRound: meta["run_context"] = &RunContext{...}  ← 存入 *RunContext
HandleInput: v = metadata["run_context"]              ← 取出 *RunContext
            runContext, _ = v.(string)                ← ❌ 断言为 string 失败
            // runContext = "" (零值)
```

**Python 参考**：Python 无类型断言，`event.metadata.get("run_context", None)` 直接获取 RunContext 对象。

**修复**：
```go
// handler.go 变量声明
var taskID string
var runKind string
var runContext *agentinterfaces.RunContext

// handler.go 类型断言
if v, ok := metadata["run_context"]; ok {
    runContext, _ = v.(*agentinterfaces.RunContext)
}

// handler.go 写入 coreTask.Metadata
if runContext != nil {
    coreTask.Metadata["run_context"] = runContext
}
```

**影响文件**：`task_loop/handler.go` L139-148, L203-205
